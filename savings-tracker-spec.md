## 1. ภาพรวมและเป้าหมาย

### สถานการณ์
คุณ (Admin) รับฝากเงินของเพื่อน 1 คนไว้ในบัญชีเงินฝากออมทรัพย์ของธนาคารเกียรตินาคินภัทร (KKP) เพื่อนโอนเข้ามาเป็นประจำ ปัจจุบันไม่มีระบบกลางที่ทั้งสองฝ่ายเห็นตรงกันว่า "ตอนนี้ยอดเท่าไหร่ ดอกเบี้ยได้เท่าไหร่ โตขึ้นแค่ไหน"

### สิ่งที่แอปนี้ต้องแก้
| ปัญหา | ทางแก้ในแอป |
|---|---|
| เพื่อนไม่รู้ยอดตัวเอง ต้องถามทุกครั้ง | Dashboard ที่เพื่อนเปิดดูได้ตลอด 24 ชม. |
| ต้องมานั่งจดว่าโอนมาเมื่อไหร่ เท่าไหร่ | แนบสลิป → ระบบอ่านและบันทึกให้อัตโนมัติ |
| ไม่รู้ว่าดอกเบี้ยควรได้เท่าไหร่ ตรวจสอบยาก | เครื่องคำนวณดอกเบี้ยตามประกาศ KKP + เทียบกับยอดจริง |
| ออมแล้วไม่รู้สึกว่าคืบหน้า เลยไม่อยากออมต่อ | เป้าหมาย + progress + streak + badge + กราฟพยากรณ์ |
| ความไว้ใจ — เงินอยู่ในบัญชีคนอื่น | ประวัติทุกรายการโปร่งใส แก้ไขได้แต่มี audit log ให้เพื่อนเห็น |

### หลักการออกแบบ 4 ข้อ
1. **โปร่งใสเป็นค่าเริ่มต้น** — เพื่อนเห็นทุกอย่างที่เกี่ยวกับเงินตัวเอง รวมถึงประวัติการแก้ไขของ Admin
2. **ตัวเลขต้องตรงกับธนาคารเสมอ** — แอปไม่ใช่แหล่งความจริง ธนาคารต่างหากที่เป็น แอปแค่ติดตามและเตือนเมื่อไม่ตรง
3. **กรอกน้อยที่สุด** — เพื่อนแค่ลากสลิปมาวาง ที่เหลือเป็นหน้าที่ของระบบ
4. **ฟรีจริง** — ทุกอย่างอยู่ในโควตาฟรีถาวร ไม่มีบัตรเครดิตผูก ไม่มี trial ที่จะหมดอายุ

## 2. ผู้ใช้และสิทธิ์

### บทบาท

**Admin (คุณ)**
- บันทึก / แก้ไข / ลบ รายการฝาก-ถอนได้ทุกรายการ
- ตั้งค่าอัตราดอกเบี้ยและเงื่อนไขบัญชี
- บันทึกดอกเบี้ยจริงที่ธนาคารจ่ายเข้ามา
- ตั้งเป้าหมายร่วมกับเพื่อน
- ดู audit log ทั้งหมด
- Export ข้อมูลเป็น CSV

**Friend (เพื่อน)**
- อัปโหลดสลิป → **ระบบอ่านและอัปเดตยอดทันที ไม่ต้องรอ Admin อนุมัติ**
- แก้ไขรายการที่ **ตัวเองสร้างเองภายใน 24 ชม.** (กันกรณีระบบอ่านผิดแล้วแก้ไม่ได้)
- ดู Dashboard, กราฟ, ประวัติ, badge ทั้งหมด
- ตั้ง/ปรับเป้าหมายเงินออมของตัวเอง
- **ลบรายการไม่ได้** — ทำได้แค่ขอยกเลิก ซึ่งจะขึ้นเป็น flag ให้ Admin เห็น

### ตารางสิทธิ์

| การกระทำ | Admin | Friend |
|---|:---:|:---:|
| อัปโหลดสลิป + บันทึกยอดทันที | ✅ | ✅ |
| แก้รายการของตัวเอง (ภายใน 24 ชม.) | ✅ | ✅ |
| แก้รายการย้อนหลังเกิน 24 ชม. | ✅ | ❌ |
| ลบรายการ | ✅ | ❌ (ขอยกเลิกได้) |
| บันทึกรายการถอน | ✅ | ❌ |
| บันทึกดอกเบี้ยที่ได้รับจริง | ✅ | ❌ |
| ตั้งค่าอัตราดอกเบี้ย | ✅ | ❌ |
| ตั้ง/แก้เป้าหมาย | ✅ | ✅ |
| ดู Dashboard และประวัติทั้งหมด | ✅ | ✅ |
| ดู audit log | ✅ | ✅ (อ่านอย่างเดียว) |

> **หมายเหตุสำคัญ:** การที่เพื่อนอัปเดตยอดได้ทันทีทำให้ประสบการณ์ลื่นไหล แต่แลกมากับความเสี่ยงว่าตัวเลขอาจผิดโดยไม่มีใครตรวจ ระบบจึงต้องมี **"ตัวถ่วงดุล"** 3 อย่าง คือ (ก) Auto-flag เมื่อ OCR มั่นใจต่ำหรือยอดผิดปกติ (ข) การกระทบยอดรายเดือนกับ statement จริง (ค) audit log ที่ทั้งคู่เห็น รายละเอียดในหัวข้อ 5 และ 15

## 3. Flow การทำงานหลัก

### Flow A — เพื่อนฝากเงิน (ทางหลัก)

```
เพื่อนโอนเงินเข้าบัญชี KKP ของ Admin ผ่านแอปธนาคาร
        ↓
เพื่อนเปิดเว็บแอป → กดปุ่ม "＋ แนบสลิป" → เลือก/ลากรูปสลิป
        ↓
[ทำงานในเบราว์เซอร์ ไม่ส่งขึ้นเซิร์ฟเวอร์]
  1. อ่าน QR code บนสลิป → ได้ transaction reference
  2. OCR ภาพ → ดึง จำนวนเงิน / วันที่-เวลา / ชื่อผู้รับ / ธนาคาร
        ↓
แสดงหน้ายืนยัน: "อ่านได้ 5,000 บาท · 18 ก.ค. 2569 14:32 · ถูกต้องไหม?"
  ├─ ถูก → กดยืนยัน
  └─ ผิด → แก้ตัวเลขเอง แล้วกดยืนยัน
        ↓
ระบบตรวจซ้ำ: มี ref นี้ / hash รูปนี้ ในระบบแล้วหรือยัง?
  ├─ ซ้ำ → ปฏิเสธ "สลิปนี้เคยบันทึกแล้วเมื่อ ..."
  └─ ไม่ซ้ำ → บันทึกเข้าตาราง transactions สถานะ 'confirmed'
        ↓
ยอดรวมอัปเดตทันที · streak +1 (ถ้าเข้าเงื่อนไข) · เช็ค badge ใหม่
        ↓
แสดง Animation ฉลอง + ยอดใหม่ + ระยะทางถึงเป้าหมาย
        ↓
Admin ได้รับ in-app notification (ไม่บล็อกอะไร แค่ให้รู้)
```

### Flow B — Admin บันทึกเอง
Admin กด "＋ เพิ่มรายการ" → เลือกประเภท (ฝาก / ถอน / ดอกเบี้ย / ปรับปรุง) → กรอกยอดและวันที่ → แนบสลิป (ถ้ามี) → บันทึก

### Flow C — ดอกเบี้ยเข้าบัญชี
```
ถึงรอบจ่ายดอกเบี้ยของธนาคาร (ดูหัวข้อ 6)
        ↓
ระบบแสดงแบนเนอร์: "ถึงรอบดอกเบี้ยแล้ว — ระบบประเมินว่าน่าจะได้ 1,234.56 บาท
                    กรุณาเช็ค statement แล้วกรอกยอดจริง"
        ↓
Admin เปิดแอป KKP → ดูยอดดอกเบี้ยจริง → กรอกลงระบบ
        ↓
ระบบเทียบ ประเมิน vs จริง
  ├─ ต่างไม่เกิน 1% → ✅ ปกติ
  └─ ต่างเกิน 1% → ⚠️ เตือน "อัตราดอกเบี้ยที่ตั้งไว้อาจไม่ตรงกับประกาศธนาคารแล้ว"
```

### Flow D — กระทบยอดรายเดือน (Reconciliation)
ทุกวันที่ 1 ระบบเตือน Admin ให้เปิดแอปธนาคาร ดูยอดคงเหลือจริง แล้วกรอกลงช่อง "ยอดจริงในบัญชี ณ สิ้นเดือน" → ระบบเทียบกับยอดในแอป → ถ้าไม่ตรง แสดงส่วนต่างพร้อมให้บันทึกรายการปรับปรุง

## 4. เรื่องการเชื่อม API ธนาคาร KKP — ความจริงที่ต้องรู้ก่อน

**สรุปสั้น: เชื่อมตรงไม่ได้ครับ และไม่ควรพยายาม**

### ทำไมถึงเชื่อมไม่ได้

| ประเด็น | รายละเอียด |
|---|---|
| ไม่มี Open Banking API สำหรับบุคคล | ประเทศไทยยังไม่มีกฎหมายบังคับให้ธนาคารเปิด API ให้เจ้าของบัญชีดึงข้อมูลตัวเอง (ต่างจาก PSD2 ของยุโรป) |
| API ที่ KKP มี เป็นระดับองค์กร | เป็นบริการอย่าง Bulk Payment / Payment Gateway / Direct Debit ที่ต้องเป็น **นิติบุคคล** ทำสัญญากับธนาคาร มีค่าธรรมเนียม และผ่านการตรวจสอบระบบ |
| การใช้ credential ส่วนตัวไปดึงข้อมูล (scraping) | **ผิดเงื่อนไขการใช้บริการของธนาคาร** และถ้าเกิดความเสียหาย ธนาคารมีสิทธิ์ปฏิเสธความรับผิด — ไม่แนะนำเด็ดขาด |
| แอป KKP Better / Dime! ไม่เปิด API สาธารณะ | ไม่มีเอกสาร ไม่มี sandbox สำหรับนักพัฒนาทั่วไป |

### ทางเลือกที่ทำได้จริง (เรียงตามที่แนะนำ)

| # | วิธี | ค่าใช้จ่าย | ความแม่นยำ | ข้อจำกัด | ใช้ในโปรเจกต์นี้ |
|---|---|---|---|---|:---:|
| 1 | **อ่าน QR + OCR สลิปในเบราว์เซอร์** | ฟรีถาวร | 85–95% | ยืนยันความแท้ของสลิปไม่ได้ | ✅ **ทางหลัก** |
| 2 | **กรอกมือ** | ฟรี | 100% (ถ้าไม่พิมพ์ผิด) | ต้องพิมพ์เอง | ✅ **ทางสำรอง** |
| 3 | **นำเข้าไฟล์ Statement (CSV/PDF)** | ฟรี | 100% | ต้องดาวน์โหลดเองเป็นรอบ ๆ | ✅ **ใช้กระทบยอด** |
| 4 | API ตรวจสอบสลิป (SlipOK, EasySlip ฯลฯ) | มีโควตาฟรีจำกัด แล้วเสียเงิน | 99% + ยืนยันความแท้ได้ | เกินโควตาต้องจ่าย ผิดเงื่อนไข "ฟรี" | ❌ (ระบุเป็น optional ไว้เฉย ๆ) |
| 5 | Bank API โดยตรง | ต้องเป็นนิติบุคคล + ค่าธรรมเนียม | 100% | เป็นไปไม่ได้ในกรณีนี้ | ❌ |
| 6 | Scraping / automation แอปธนาคาร | ฟรี | ไม่แน่นอน | ผิดเงื่อนไข เสี่ยงบัญชีถูกระงับ | ❌ **ห้ามทำ** |

### สถาปัตยกรรมที่เลือก
ออกแบบให้ **layer การนำเข้าข้อมูลเป็นแบบถอดเปลี่ยนได้ (pluggable)** — เขียน interface กลางชื่อ `SlipParser` ไว้ วันหนึ่งถ้าอยากอัปเกรดไปใช้ API ที่มีค่าใช้จ่าย ก็แค่เขียน adapter ตัวใหม่มาเสียบ ไม่ต้องรื้อระบบ

```ts
interface SlipParser {
  name: string
  parse(file: File): Promise<{
    amount: number | null
    datetime: Date | null
    reference: string | null
    receiverName: string | null
    senderName: string | null
    confidence: number      // 0.0 – 1.0
    verified: boolean       // true เฉพาะเมื่อยืนยันกับธนาคารได้จริง
  }>
}
// v1: LocalOcrParser (ฟรี)  →  อนาคต: SlipVerifyApiParser (เสียเงิน)
```

## 5. ระบบอ่านสลิปอัตโนมัติ (ฟรี 100%)

### สถาปัตยกรรม: ทำงานฝั่งเบราว์เซอร์ทั้งหมด
ไม่ส่งรูปไปประมวลผลบนเซิร์ฟเวอร์ใคร → ฟรี, เร็ว, และข้อมูลการเงินไม่รั่วไปที่สาม

### ขั้นตอนที่ 1 — อ่าน QR Code
สลิปโอนเงินของธนาคารไทยเกือบทุกแห่งมี QR เล็ก ๆ (mini-QR) ที่บรรจุ **รหัสอ้างอิงรายการ (transaction reference)** ตามมาตรฐาน EMVCo

- ไลบรารี: `jsQR` หรือ `html5-qrcode` (MIT License, ฟรี)
- ได้อะไร: รหัสอ้างอิงที่ **ไม่ซ้ำกันทั้งประเทศ** → ใช้เป็นกุญแจกันบันทึกซ้ำได้ดีที่สุด
- ไม่ได้อะไร: QR **ไม่มีจำนวนเงิน** อยู่ในนั้น ต้องพึ่ง OCR

### ขั้นตอนที่ 2 — OCR ดึงตัวเลข
- ไลบรารี: **Tesseract.js** (Apache 2.0, ฟรี) + ชุดภาษา `tha` และ `eng`
- Pre-processing ก่อน OCR เพื่อเพิ่มความแม่น:
  1. แปลงเป็น grayscale
  2. ปรับ contrast + binarize (Otsu threshold)
  3. Auto-rotate ถ้าภาพเอียง
  4. Upscale ถ้าภาพเล็กกว่า 1000px
- Regex ที่ใช้จับข้อมูล:

```
จำนวนเงิน   /(\d{1,3}(?:,\d{3})*\.\d{2})\s*(?:บาท|THB|baht)/i
วันที่ไทย   /(\d{1,2})\s*(ม\.ค\.|ก\.พ\.|มี\.ค\.|เม\.ย\.|พ\.ค\.|มิ\.ย\.|ก\.ค\.|ส\.ค\.|ก\.ย\.|ต\.ค\.|พ\.ย\.|ธ\.ค\.)\s*(\d{2,4})/
เวลา        /(\d{1,2}):(\d{2})(?::(\d{2}))?\s*(?:น\.)?/
รหัสอ้างอิง /(?:รหัสอ้างอิง|Ref(?:erence)?(?:\s*No)?)[:\s]*([A-Z0-9]{10,})/i
```

- **แปลง พ.ศ. → ค.ศ.** อัตโนมัติ: ถ้าปีที่อ่านได้ > 2400 ให้ลบ 543

### ขั้นตอนที่ 3 — ให้คะแนนความมั่นใจ

| เงื่อนไข | คะแนน |
|---|---|
| อ่าน QR ได้ | +0.35 |
| อ่านจำนวนเงินได้ (รูปแบบถูกต้อง) | +0.30 |
| อ่านวันที่-เวลาได้ | +0.20 |
| ชื่อบัญชีผู้รับตรงกับที่ตั้งค่าไว้ | +0.15 |

**การตัดสินใจตามคะแนน:**
- `≥ 0.80` → กรอกฟอร์มให้อัตโนมัติ ผู้ใช้แค่กดยืนยัน (ทางลื่นไหล)
- `0.50 – 0.79` → กรอกให้แต่ **ไฮไลต์ช่องที่ไม่มั่นใจเป็นสีเหลือง** ให้ตรวจก่อนยืนยัน
- `< 0.50` → ให้กรอกมือทั้งหมด และติด flag `needs_review` ให้ Admin เห็นบน Dashboard

### ขั้นตอนที่ 4 — ป้องกันสลิปซ้ำ (3 ชั้น)
1. **รหัสอ้างอิง** — ถ้าซ้ำ ปฏิเสธทันที (แม่นที่สุด)
2. **SHA-256 hash ของไฟล์รูป** — จับกรณีอัปโหลดรูปเดิมซ้ำ
3. **ยอดเงิน + วันที่-เวลา (± 2 นาที)** — จับกรณีถ่ายรูปสลิปเดิมใหม่ ระบบจะถามยืนยัน "รายการนี้คล้ายกับที่บันทึกไว้เมื่อ ... ต้องการบันทึกซ้ำจริงหรือไม่?"

### ขั้นตอนที่ 5 — Auto-flag รายการผิดปกติ
ระบบติดธงให้ Admin ตรวจ (แต่ **ไม่บล็อก** การบันทึก) เมื่อ:
- ยอดสูงกว่าค่าเฉลี่ย 3 เดือนย้อนหลัง เกิน 3 เท่า
- วันที่บนสลิปย้อนหลังเกิน 7 วัน
- ความมั่นใจ OCR < 0.5
- ชื่อบัญชีผู้รับที่อ่านได้ ไม่ตรงกับที่ตั้งค่าไว้
- ยอดเป็นเลขกลม ๆ ผิดปกติเมื่อเทียบกับพฤติกรรมเดิม

### เก็บไฟล์สลิป
- อัปโหลดขึ้น Supabase Storage (bucket ชื่อ `slips`, ตั้งเป็น **private**)
- บีบอัดเป็น WebP คุณภาพ 80 ก่อนอัปโหลด → ประมาณ 80–150 KB ต่อรูป
- โควตาฟรี 1 GB ≈ เก็บได้ **7,000+ สลิป** (ฝากเดือนละ 2 ครั้ง = ใช้ได้ 290 ปี)
- เข้าถึงผ่าน signed URL อายุ 60 วินาทีเท่านั้น

## 6. เครื่องคำนวณดอกเบี้ย — ส่วนที่ต้องแม่นที่สุด

> ⚠️ **หลักการสำคัญ:** แอปนี้ **ประเมิน** ดอกเบี้ย ไม่ใช่ **กำหนด** ดอกเบี้ย ตัวเลขจริงคือสิ่งที่ธนาคารจ่ายเข้าบัญชี ระบบจึงต้องออกแบบให้ตัวเลขประเมินเทียบกับตัวเลขจริงได้เสมอ และเตือนเมื่อไม่ตรง

### 6.1 หลักการคำนวณของบัญชีออมทรัพย์ไทย

ธนาคารไทยคำนวณดอกเบี้ยออมทรัพย์แบบ **สะสมรายวันจากยอดคงเหลือสิ้นวัน** แล้วจ่ายเป็นรอบ

```
ดอกเบี้ยของวันนั้น = ยอดคงเหลือสิ้นวัน × อัตราดอกเบี้ยเฉลี่ยต่อปี ÷ 365
```

กรณีอัตราเป็นแบบ **ขั้นบันได (tiered)** ต้องคำนวณแยกแต่ละขั้นก่อน แล้วรวมกัน:

```
ดอกเบี้ยของวันนั้น = Σ (ยอดในขั้นที่ i × อัตราขั้นที่ i) ÷ 365
```

**ตัวอย่าง** — ยอดคงเหลือ 350,000 บาท กับโครงสร้างอัตราสมมติ:

| ขั้น | ช่วงยอดเงิน | อัตรา/ปี | ยอดในขั้นนี้ | ดอกเบี้ย/ปี |
|---|---|---|---|---|
| 1 | 0 – 10,000 | 3.00% | 10,000 | 300.00 |
| 2 | 10,001 – 200,000 | 1.50% | 190,000 | 2,850.00 |
| 3 | 200,001 – 2,000,000 | 1.55% | 150,000 | 2,325.00 |
| | | | **รวม** | **5,475.00** |

→ ดอกเบี้ยวันนั้น = 5,475 ÷ 365 = **15.00 บาท**
→ อัตราเฉลี่ยที่แท้จริง = 5,475 ÷ 350,000 = **1.564%**

### 6.2 ตารางอัตราดอกเบี้ยต้องเป็นข้อมูล ไม่ใช่โค้ด

**ห้าม hardcode อัตราดอกเบี้ยในโค้ดเด็ดขาด** เพราะธนาคารปรับอัตราปีละหลายครั้ง ให้เก็บในฐานข้อมูลพร้อม **วันที่มีผลบังคับใช้** เพื่อให้คำนวณย้อนหลังได้ถูกต้อง

**ต้องบันทึกเงื่อนไขเหล่านี้ด้วย** (ดูจากประกาศธนาคาร):
- ยอดเงินฝากขั้นต่ำที่ได้รับอัตราพิเศษ
- อัตราพิเศษมีเงื่อนไขจำนวนครั้งการถอนต่อเดือนหรือไม่
- อัตราพิเศษเป็นโปรโมชันที่มีวันหมดอายุหรือไม่ (สำคัญมาก — หลายบัญชีเป็นแบบนี้)
- รอบการจ่ายดอกเบี้ย (ออมทรัพย์ทั่วไปมักปีละ 2 ครั้ง ราวสิ้นเดือน มิ.ย. และ ธ.ค. แต่บางบัญชีจ่ายรายเดือน — **ต้องเช็คของบัญชีตัวเอง**)

### 6.3 ภาษีหัก ณ ที่จ่าย
- ดอกเบี้ยเงินฝากออมทรัพย์ของบุคคลธรรมดา หากรวมทุกบัญชีทุกธนาคารในปีภาษี **ไม่เกิน 20,000 บาท** จะได้รับยกเว้นภาษี
- หากเกิน จะถูกหักภาษี ณ ที่จ่าย **15%** ของดอกเบี้ยทั้งจำนวน (ไม่ใช่เฉพาะส่วนที่เกิน)
- ระบบต้องมีสวิตช์เปิด/ปิดการคำนวณภาษี และแสดงทั้ง **ดอกเบี้ยก่อนหักภาษี** และ **สุทธิ**
- แสดงมาตรวัดเตือนเมื่อดอกเบี้ยสะสมในปีเข้าใกล้ 20,000 บาท

> ⚠️ เงินก้อนนี้อยู่ในบัญชีของคุณ (Admin) ดังนั้น **ภาระภาษีเป็นของคุณ** ไม่ใช่ของเพื่อน ควรตกลงกันให้ชัดว่าใครรับผิดชอบส่วนนี้ และให้ระบบบันทึกข้อตกลงไว้ในหน้าตั้งค่า

### 6.4 อัลกอริทึมการคำนวณ

```ts
function calcDailyInterest(balance: number, tiers: RateTier[]): number {
  let interest = 0
  for (const t of tiers) {
    const upper = t.max_balance ?? Infinity
    const amountInTier = Math.max(0, Math.min(balance, upper) - t.min_balance)
    if (amountInTier <= 0) continue
    interest += (amountInTier * t.annual_rate) / 365
  }
  return interest
}

function accrueInterest(txns: Txn[], from: Date, to: Date): DailyAccrual[] {
  const result: DailyAccrual[] = []
  let balance = balanceAsOf(txns, from)
  for (const day of eachDay(from, to)) {
    balance += netMovementOn(txns, day)          // ยอดคงเหลือ ณ สิ้นวัน
    const tiers = ratesEffectiveOn(day)           // ใช้อัตราที่มีผลในวันนั้น
    result.push({ date: day, balance, interest: calcDailyInterest(balance, tiers) })
  }
  return result
}
```

**ข้อควรระวังในการเขียนโค้ด:**
- ใช้ **จำนวนเต็มหน่วยสตางค์** (integer cents) ในการคำนวณทั้งหมด ห้ามใช้ float กับเงิน
- ปัดเศษเฉพาะตอน **แสดงผล** และตอน **ลงบัญชีจริง** เท่านั้น การสะสมรายวันให้เก็บทศนิยมเต็ม
- ใช้ 365 วันเสมอ แม้ปีอธิกสุรทิน (เป็นวิธีมาตรฐานของธนาคารไทย) — แต่ควรทำเป็นค่าตั้งค่าเผื่อไว้
- ใช้ timezone `Asia/Bangkok` ในการตัดวัน

### 7.4 กราฟพยากรณ์ถึงเป้าหมาย

**การคำนวณ** — ใช้ค่าเฉลี่ยการฝากถ่วงน้ำหนัก 6 เดือนล่าสุด (เดือนล่าสุดมีน้ำหนักมากกว่า) แล้วฉายอนาคตพร้อมทบดอกเบี้ย

```ts
function forecastMonthsToGoal(current: number, goal: number,
                              avgMonthlyDeposit: number, annualRate: number): number {
  const r = annualRate / 12
  let balance = current, months = 0
  while (balance < goal && months < 600) {
    balance = balance * (1 + r) + avgMonthlyDeposit
    months++
  }
  return months
}
```

**แสดงผลเป็นกราฟเส้น 3 เส้น**
- 🔵 **เส้นทึบ (อดีต)** — ยอดจริงที่ผ่านมา
- 🔵 **เส้นประ (คาดการณ์)** — ต่อยอดจากพฤติกรรมปัจจุบัน
- 🟢 **เส้นเป้าหมาย** — เส้นแนวนอนที่ยอดเป้าหมาย พร้อมหมุดวันที่ตั้งใจ

**ข้อความใต้กราฟ (ส่วนที่กระตุ้นที่สุด)**
> ถ้าฝากเดือนละ **8,500 บาท** เท่าเดิม จะถึงเป้าหมาย 500,000 บาท ใน **เดือนมีนาคม 2571** (อีก 20 เดือน)
> 💡 ถ้าเพิ่มเป็นเดือนละ **10,000 บาท** จะถึงเป้าเร็วขึ้น **4 เดือน** และประหยัดเวลารอไปได้ **1 ใน 5**

พร้อม slider ให้ลากเล่นว่า "ถ้าฝากเดือนละ X บาท จะถึงเป้าเมื่อไหร่" — การได้เห็นตัวเลขขยับตามมือตัวเองคือแรงกระตุ้นที่ทรงพลังที่สุด

## 9. โครงสร้างฐานข้อมูล

```sql
-- ผู้ใช้ (ผูกกับ auth.users ของ Supabase)
create table profiles (
  id          uuid primary key references auth.users(id),
  display_name text not null,
  role        text not null check (role in ('admin','friend')),
  created_at  timestamptz default now()
);

-- บัญชีเงินฝาก
create table accounts (
  id             uuid primary key default gen_random_uuid(),
  name           text not null,              -- 'KKP Savvy — เงินฝากของ [ชื่อเพื่อน]'
  bank           text default 'KKP',
  account_type   text,                       -- 'savings'
  receiver_name  text,                       -- ชื่อบนสลิป ใช้ตรวจสอบ
  owner_id       uuid references profiles(id), -- Admin
  beneficiary_id uuid references profiles(id), -- เพื่อน
  interest_payout text default 'semiannual', -- monthly | quarterly | semiannual
  tax_enabled    boolean default false,
  created_at     timestamptz default now()
);

-- รายการเคลื่อนไหว (หัวใจของระบบ)
create table transactions (
  id             uuid primary key default gen_random_uuid(),
  account_id     uuid references accounts(id) not null,
  type           text not null check (type in ('deposit','withdrawal','interest','adjustment')),
  amount_cents   bigint not null,            -- หน่วยสตางค์ ห้ามใช้ float
  occurred_at    timestamptz not null,       -- เวลาบนสลิป ไม่ใช่เวลาบันทึก
  note           text,
  slip_path      text,                       -- path ใน Supabase Storage
  slip_hash      text,                       -- SHA-256 กันซ้ำ
  bank_reference text unique,                -- รหัสอ้างอิงจาก QR กันซ้ำ
  ocr_confidence numeric(3,2),
  ocr_raw        jsonb,                      -- เก็บผลดิบไว้ debug
  needs_review   boolean default false,
  created_by     uuid references profiles(id),
  created_at     timestamptz default now(),
  edited_at      timestamptz,
  cancel_requested boolean default false
);
create index on transactions (account_id, occurred_at desc);

-- อัตราดอกเบี้ยขั้นบันได (มีวันที่มีผล → คำนวณย้อนหลังได้ถูก)
create table interest_rate_tiers (
  id             uuid primary key default gen_random_uuid(),
  account_id     uuid references accounts(id) not null,
  effective_from date not null,
  effective_to   date,                       -- null = ยังใช้อยู่
  tier_order     int not null,
  min_balance    numeric(15,2) not null,
  max_balance    numeric(15,2),              -- null = ไม่จำกัด
  annual_rate    numeric(6,5) not null,      -- 0.0155 = 1.555%
  source_note    text,                       -- อ้างอิงประกาศธนาคาร
  source_url     text
);

-- ดอกเบี้ยสะสมรายวัน (คำนวณล่วงหน้าเก็บไว้เพื่อความเร็ว)
create table interest_accruals (
  account_id     uuid references accounts(id),
  accrual_date   date not null,
  closing_balance numeric(15,2) not null,
  effective_rate numeric(6,5) not null,
  interest       numeric(15,6) not null,     -- เก็บทศนิยม 6 ตำแหน่ง ไม่ปัด
  primary key (account_id, accrual_date)
);

-- รอบการจ่ายดอกเบี้ย (เทียบประเมิน vs จริง)
create table interest_periods (
  id              uuid primary key default gen_random_uuid(),
  account_id      uuid references accounts(id),
  period_start    date not null,
  period_end      date not null,
  estimated_gross numeric(15,2),
  actual_gross    numeric(15,2),
  tax_withheld    numeric(15,2) default 0,
  actual_net      numeric(15,2),
  variance_pct    numeric(6,2) generated always as
                  (case when estimated_gross > 0
                        then (actual_gross - estimated_gross) / estimated_gross * 100
                        end) stored,
  recorded_by     uuid references profiles(id),
  recorded_at     timestamptz
);

-- เป้าหมาย
create table goals (
  id            uuid primary key default gen_random_uuid(),
  account_id    uuid references accounts(id),
  title         text not null,
  target_amount numeric(15,2) not null,
  target_date   date,
  is_active     boolean default true,
  achieved_at   timestamptz,
  created_at    timestamptz default now()
);

-- Badge ที่ได้รับ
create table achievements (
  id           uuid primary key default gen_random_uuid(),
  profile_id   uuid references profiles(id),
  badge_code   text not null,
  earned_at    timestamptz default now(),
  unique (profile_id, badge_code)
);

-- กระทบยอดรายเดือน
create table reconciliations (
  id             uuid primary key default gen_random_uuid(),
  account_id     uuid references accounts(id),
  as_of_date     date not null,
  bank_balance   numeric(15,2) not null,     -- ยอดจริงจาก statement
  system_balance numeric(15,2) not null,     -- ยอดในแอป
  difference     numeric(15,2) generated always as (bank_balance - system_balance) stored,
  resolved       boolean default false,
  note           text,
  created_at     timestamptz default now()
);

-- Audit log (append-only ห้ามแก้ห้ามลบ)
create table audit_log (
  id          bigserial primary key,
  actor_id    uuid references profiles(id),
  action      text not null,               -- create | update | delete | login
  entity      text not null,               -- transactions | goals | rates ...
  entity_id   uuid,
  before      jsonb,
  after       jsonb,
  created_at  timestamptz default now()
);
```

**Trigger ที่ต้องมี**
- `after insert/update/delete on transactions` → เขียน audit_log อัตโนมัติ
- `after insert on transactions` → คำนวณ accrual ใหม่ตั้งแต่วันที่ของรายการนั้นเป็นต้นไป
- `after insert on transactions` → ตรวจเงื่อนไข badge และ streak

## 10. ความปลอดภัย

### Row Level Security (RLS) — เปิดทุกตาราง
```sql
alter table transactions enable row level security;

-- ทั้งคู่อ่านได้ (เฉพาะบัญชีที่ตัวเองเกี่ยวข้อง)
create policy "read own account" on transactions for select
  using (account_id in (
    select id from accounts
    where owner_id = auth.uid() or beneficiary_id = auth.uid()
  ));

-- ทั้งคู่สร้างได้
create policy "insert own account" on transactions for insert
  with check (account_id in (
    select id from accounts
    where owner_id = auth.uid() or beneficiary_id = auth.uid()
  ));

-- แก้ได้: Admin แก้ได้ทุกอัน / เพื่อนแก้ได้เฉพาะของตัวเองภายใน 24 ชม.
create policy "update rules" on transactions for update
  using (
    exists (select 1 from profiles where id = auth.uid() and role = 'admin')
    or (created_by = auth.uid() and created_at > now() - interval '24 hours')
  );

-- ลบได้เฉพาะ Admin
create policy "admin delete only" on transactions for delete
  using (exists (select 1 from profiles where id = auth.uid() and role = 'admin'));
```

### เรื่องอื่น ๆ
- **Auth:** Magic Link ทางอีเมล — ไม่มีรหัสผ่านให้หลุด และ Supabase ให้ฟรี
- **Storage:** bucket `slips` ตั้ง private, เข้าถึงผ่าน signed URL อายุ 60 วินาที
- **ห้ามเก็บ:** เลขบัญชีเต็ม, เลขบัตรประชาชน, credential ธนาคาร — เก็บแค่ 4 หลักท้ายไว้อ้างอิงพอ
- **API key:** ใช้เฉพาะ `anon key` ในฝั่ง frontend (ปลอดภัยเมื่อมี RLS) — **ห้าม** เอา `service_role key` ไว้ในโค้ด frontend เด็ดขาด
- **Backup:** ตั้ง GitHub Action รันสัปดาห์ละครั้ง export ข้อมูลเป็น CSV เก็บใน private repo (ฟรี) เพราะแพลนฟรีของ Supabase มี point-in-time recovery จำกัด

## 13. คู่มือติดตั้งและ Deploy แบบทีละขั้น

### สิ่งที่ต้องมีก่อน
- Node.js เวอร์ชัน 20 ขึ้นไป ([nodejs.org](https://nodejs.org))
- บัญชี GitHub, Supabase, Vercel (สมัครฟรีทั้งหมด ใช้ GitHub ล็อกอินได้)

### ขั้นที่ 1 — สร้างโปรเจกต์
```bash
npm create vite@latest savings-tracker -- --template react-ts
cd savings-tracker
npm install
npm install @supabase/supabase-js recharts date-fns date-fns-tz tesseract.js jsqr
npm install -D tailwindcss @tailwindcss/vite
```

### ขั้นที่ 2 — สร้างโปรเจกต์ Supabase
1. เข้า supabase.com → New Project → เลือก region **Southeast Asia (Singapore)** (ใกล้ไทยที่สุด ทำให้เร็ว)
2. ตั้งรหัสผ่านฐานข้อมูล → **จดเก็บไว้ดี ๆ**
3. รอประมาณ 2 นาที
4. ไปที่ **SQL Editor** → วาง SQL จากหัวข้อ 9 ทั้งหมด → กด Run
5. ไปที่ **Storage** → New bucket ชื่อ `slips` → ตั้งเป็น **Private**
6. ไปที่ **Settings → API** → คัดลอก `Project URL` และ `anon public key`

### ขั้นที่ 3 — ตั้งค่า environment
สร้างไฟล์ `.env.local` ที่ root ของโปรเจกต์
```
VITE_SUPABASE_URL=https://xxxxx.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGci...
```
เพิ่ม `.env.local` ลงใน `.gitignore` (ปกติมีให้อยู่แล้ว — **เช็คให้แน่ใจ**)

### ขั้นที่ 4 — ทดสอบในเครื่อง
```bash
npm run dev
# เปิด http://localhost:5173
```

### ขั้นที่ 5 — ขึ้น GitHub
```bash
git init
git add .
git commit -m "initial"
gh repo create savings-tracker --private --source=. --push
```
> ถ้าไม่มีคำสั่ง `gh` ให้สร้าง repo แบบ **Private** ผ่านหน้าเว็บ GitHub แล้วทำตามคำสั่งที่หน้านั้นแสดง

### ขั้นที่ 6 — Deploy บน Vercel
1. เข้า vercel.com → Add New Project → เลือก repo ที่เพิ่งสร้าง
2. Framework Preset: **Vite** (ตรวจจับให้อัตโนมัติ)
3. ไปที่ **Environment Variables** → ใส่ `VITE_SUPABASE_URL` และ `VITE_SUPABASE_ANON_KEY`
4. กด Deploy → รอ 1–2 นาที → ได้ URL เช่น `savings-tracker.vercel.app`

### ขั้นที่ 7 — เพิ่มผู้ใช้ 2 คน
1. Supabase → **Authentication → Users → Add user** → ใส่อีเมลของคุณ และของเพื่อน
2. Supabase → **SQL Editor** → รันคำสั่งกำหนดบทบาท:
```sql
insert into profiles (id, display_name, role) values
  ('<uuid-ของคุณ>',    'ชื่อคุณ',   'admin'),
  ('<uuid-ของเพื่อน>', 'ชื่อเพื่อน', 'friend');
```
3. Supabase → **Authentication → URL Configuration** → ใส่ URL ของ Vercel ลงใน Redirect URLs (ไม่งั้น Magic Link จะเด้งกลับไม่ถูก)

### ขั้นที่ 8 — ตั้งค่าเริ่มต้น
1. เข้าเว็บด้วยบัญชี Admin
2. หน้าตั้งค่า → สร้างบัญชี ระบุชื่อผู้รับที่ปรากฏบนสลิป
3. **หน้าตั้งค่าอัตราดอกเบี้ย → เปิด PDF ประกาศของ KKP กรอกขั้นบันไดตามบัญชีจริง**
4. บันทึกยอดตั้งต้น (ยอดที่มีอยู่แล้ว ณ วันนี้) เป็นรายการประเภท `adjustment`
5. ตั้งเป้าหมายร่วมกับเพื่อน
6. ส่งลิงก์ให้เพื่อน → เพื่อนกรอกอีเมล → รับ Magic Link → เข้าใช้งานได้เลย

### ขั้นที่ 9 — ตั้ง Backup อัตโนมัติ (แนะนำ)
สร้างไฟล์ `.github/workflows/backup.yml` ให้รันทุกวันอาทิตย์ ดึงข้อมูลจาก Supabase ออกมาเป็น CSV แล้ว commit เข้า repo — ทำหน้าที่ทั้ง backup และ keep-alive กันโปรเจกต์ถูก pause ไปในตัว

## 14. คู่มือการใช้งาน

### สำหรับเพื่อน (ผู้ฝาก)

**ฝากเงิน**
1. โอนเงินเข้าบัญชีตามปกติผ่านแอปธนาคาร
2. บันทึกภาพหน้าจอสลิปไว้
3. เปิดเว็บแอป → กดปุ่ม **＋** สีเขียวมุมขวาล่าง
4. เลือกรูปสลิป → รอ 2–5 วินาที
5. ตรวจตัวเลขที่ระบบอ่านให้ → ถ้าถูกกด **ยืนยัน** ถ้าผิดแก้ก่อนแล้วค่อยยืนยัน
6. เสร็จ! ยอดขึ้นทันที

**ถ้าระบบอ่านสลิปไม่ออก** — ไม่ต้องตกใจ แค่กรอกจำนวนเงินและวันที่เอง รูปสลิปยังถูกเก็บไว้เป็นหลักฐานเหมือนเดิม

**แก้ไขรายการที่พลาด** — เข้าหน้าประวัติ → กดที่รายการ → แก้ได้ภายใน 24 ชม. หลังจากนั้นต้องกด "ขอยกเลิก" แล้วแจ้ง Admin

**ดูความคืบหน้า** — หน้า Dashboard คือหน้าแรกที่เจอ ดูเป้าหมายและกราฟพยากรณ์ที่หน้า "เป้าหมาย" ลอง slider เล่นดูว่าถ้าฝากเพิ่มอีกนิดจะถึงเป้าเร็วขึ้นแค่ไหน

**เพิ่มลงหน้าจอโฮม** — เปิดเว็บบนมือถือ → เมนูเบราว์เซอร์ → "เพิ่มไปยังหน้าจอโฮม" → ใช้เหมือนแอปจริง

### สำหรับคุณ (Admin)

**งานประจำเดือน (ใช้เวลา ~5 นาที)**
1. วันที่ 1 ของเดือน ระบบจะเตือนให้กระทบยอด
2. เปิดแอป KKP → ดูยอดคงเหลือจริง
3. กรอกลงหน้า "กระทบยอด" → ระบบเทียบกับยอดในแอปให้
4. ถ้าตรง ✅ จบ / ถ้าไม่ตรง ดูส่วนต่างแล้วหาสาเหตุ (มักเป็นรายการที่ลืมบันทึก หรือค่าธรรมเนียม)

**งานเมื่อดอกเบี้ยเข้า (ปีละ 1–2 ครั้ง)**
1. ระบบขึ้นแบนเนอร์แจ้งว่าถึงรอบแล้ว
2. เปิด statement ดูยอดดอกเบี้ยจริงและภาษีที่ถูกหัก
3. กรอกลงหน้า "ดอกเบี้ย"
4. ถ้าระบบเตือนว่าต่างเกิน 1% → ไปเช็คประกาศอัตราดอกเบี้ยล่าสุดของธนาคาร แล้วอัปเดตในหน้าตั้งค่า

**เมื่อธนาคารปรับอัตราดอกเบี้ย**
1. หน้าตั้งค่าอัตราดอกเบี้ย → กด "เพิ่มชุดอัตราใหม่"
2. ระบุ **วันที่มีผล** ให้ตรงกับประกาศธนาคาร
3. กรอกขั้นบันไดใหม่ + ใส่ลิงก์ประกาศไว้ในช่องอ้างอิง
4. ระบบจะคำนวณย้อนหลังใหม่โดยใช้อัตราที่ถูกต้องของแต่ละช่วงเวลาให้เอง — **ห้ามแก้ทับชุดเดิม** เพราะจะทำให้การคำนวณย้อนหลังผิด

**ตรวจรายการที่ติดธง** — Dashboard จะขึ้นตัวเลขสีส้มบอกจำนวนรายการที่ระบบไม่มั่นใจ กดเข้าไปดูรูปสลิปเทียบกับตัวเลขได้เลย

## 15. ความเสี่ยงและวิธีรับมือ

| ความเสี่ยง | ระดับ | วิธีรับมือ |
|---|:---:|---|
| **สลิปปลอม / ตัวเลขไม่ตรงกับที่โอนจริง** | 🔴 สูง | ระบบฟรีไม่สามารถยืนยันความแท้ของสลิปกับธนาคารได้ → พึ่ง **การกระทบยอดรายเดือนกับ statement จริง** เป็นตัวจับ ถ้ายอดในแอปไม่ตรงกับธนาคาร แปลว่ามีอะไรผิดแน่นอน |
| **OCR อ่านผิด แล้วเพื่อนกดยืนยันโดยไม่ดู** | 🟠 กลาง | ไฮไลต์ช่องที่ไม่มั่นใจเป็นสีเหลือง + auto-flag ให้ Admin ตรวจ + กระทบยอดรายเดือนจับได้อยู่ดี |
| **อัตราดอกเบี้ยในระบบล้าสมัย** | 🟠 กลาง | เตือนอัตโนมัติเมื่อดอกเบี้ยจริงต่างจากประเมินเกิน 1% + เตือนทุก 6 เดือนให้เช็คประกาศธนาคาร |
| **โปรโมชันอัตราพิเศษหมดอายุโดยไม่รู้ตัว** | 🟠 กลาง | บันทึกวันหมดอายุโปรโมชันในตารางอัตรา + แจ้งเตือนล่วงหน้า 30 วัน |
| **ข้อมูลหาย** | 🟡 ต่ำ | Backup อัตโนมัติสัปดาห์ละครั้งขึ้น GitHub private repo + ปุ่ม export CSV ให้กดเองได้ตลอด |
| **Supabase pause เพราะไม่มีคนใช้** | 🟡 ต่ำ | GitHub Action ping สัปดาห์ละครั้ง (และข้อมูลไม่หายอยู่แล้ว แค่ต้องกดปลุก) |
| **ความสัมพันธ์เสียเพราะเรื่องเงิน** | 🔴 สูง | นี่คือความเสี่ยงที่แอปแก้ไม่ได้ทั้งหมด → ทำให้ทุกอย่างโปร่งใสที่สุด (audit log ที่เพื่อนเห็น, ทุกรายการมีสลิป, กระทบยอดสม่ำเสมอ) และ **ควรมีข้อตกลงเป็นลายลักษณ์อักษรนอกแอป** |

### 🔔 สิ่งที่ควรตกลงกันให้ชัดก่อนเริ่มใช้ (นอกเหนือจากตัวแอป)

แอปนี้ช่วยเรื่องการติดตามตัวเลข แต่ไม่ได้ทำให้ข้อตกลงระหว่างคนสองคนชัดขึ้น ประเด็นที่ควรคุยกันและบันทึกไว้ในหน้า "ตั้งค่า → ข้อตกลง":

1. **ดอกเบี้ยเป็นของใคร** — ทั้งหมดเป็นของเพื่อน, แบ่งกัน, หรือคุณเก็บไว้เป็นค่าดูแล?
2. **ภาษีหัก ณ ที่จ่าย 15% ใครรับผิดชอบ** — เงินอยู่ในชื่อคุณ ภาระภาษีจึงตกที่คุณตามกฎหมาย
3. **ถอนได้เมื่อไหร่ แจ้งล่วงหน้ากี่วัน** — และจะถอนบางส่วนได้ไหม
4. **ถ้ามีเหตุไม่คาดฝัน** — เกิดอะไรขึ้นกับเงินก้อนนี้ ควรมีคนที่สามรู้เรื่อง
5. **ทางกฎหมาย** — เงินฝากในชื่อคุณถือเป็นทรัพย์สินของคุณตามกฎหมาย ถ้าเป็นเงินก้อนใหญ่ ควรมีหนังสือรับสภาพหนี้หรือสัญญายืมเงินเป็นลายลักษณ์อักษร (ผมไม่ใช่นักกฎหมาย — ถ้าเป็นจำนวนที่มีนัยสำคัญ ควรปรึกษาผู้เชี่ยวชาญ)

## ภาคผนวก — Checklist ก่อนเปิดใช้จริง

- [ ] กรอกอัตราดอกเบี้ยจาก **PDF ประกาศฉบับล่าสุด** ของธนาคาร ไม่ใช่จากบทความข่าว
- [ ] ยืนยันชื่อบัญชีและประเภทบัญชีที่ใช้จริง (แต่ละบัญชี KKP อัตราต่างกันมาก)
- [ ] ตรวจสอบรอบการจ่ายดอกเบี้ยของบัญชีนั้น ๆ
- [ ] บันทึกยอดตั้งต้นให้ตรงกับ statement ณ วันเริ่มใช้
- [ ] ทดสอบอัปโหลดสลิปจริง 3–5 ใบ ดูว่า OCR อ่านได้แม่นแค่ไหน
- [ ] ทดสอบ RLS โดยล็อกอินด้วยบัญชีเพื่อน แล้วลองทำสิ่งที่ไม่ควรทำได้ (ต้องถูกปฏิเสธ)
- [ ] ตั้ง backup อัตโนมัติและทดสอบว่า restore กลับได้จริง
- [ ] คุยข้อตกลง 5 ข้อข้างบนกับเพื่อนให้จบก่อน
